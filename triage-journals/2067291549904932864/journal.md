## Executive Summary (Corrected Analysis)
*Note: The initial automated analysis of this build incorrectly diagnosed the issue as CPU saturation caused by a "Thundering Herd" from the `watch-list` utility. This has been corrected based on deeper engineering analysis.*

The 5k-node scalability test failed due to an API Responsiveness SLO breach (p99 `LIST pods` latency hit 43.64s, limit 30s). The true root cause is a **Go Scheduler Wakeup Storm** driven by massive endpointslice watch fan-out. The apiserver was not CPU-bound (~80 of 96 cores were idle); instead, the Go runtime's global scheduler lock was paralyzed by hundreds of millions of goroutine wakeups.

**Classification:** Emergent System Limit (Architectural Scale Limit / Runtime Bottleneck)

## Triage Narrative & Findings

### 1. The True Bottleneck: Go Scheduler Starvation
While the initial analysis saw high lock contention and assumed the CPU was saturated, the metrics prove the apiserver is barely busy (14-16 of 96 cores in use). The Go scheduler cannot get goroutines onto cores fast enough.

Goroutines pile into the run queue faster than the runtime can place them on the idle cores because every handoff goes through a single global scheduler lock.

### 2. The Root Cause of Load: EndpointSlice Fan-out
The massive load is generated by endpointslice watch wakeups. Every single endpointslice change wakes up ~5,317 clients (a `kube-proxy` on every node + CoreDNS). 
Over a run, ~261,000 endpointslice changes turn into 680 million wakeups, accounting for 77% of all watch wakeups on the apiserver. It is the pure overhead of waking these goroutines that buries the Go scheduler.

### 3. The `LIST pods` SLO Breach is the Victim
The pod `LIST` is just the request that gets measured for the SLO. A cluster-wide pod LIST returns ~150-230MB. It gets pushed off the CPU hundreds to thousands of times while it waits in the run-queue. About 10 seconds of real work stretches to 40-50+ seconds as it repeatedly loses its timeslice to the massive queue.

### 4. The Feedback Loop
This run failed because it got stuck in the slow state for too long.
Because the apiserver was slow, the Controller Manager created pods much slower. This forced the test framework (`clusterloader2`) to wait much longer for the deployments to come up before it could start tearing them down. By delaying the teardown (which clears the watch traffic), the watches stayed active longer, sustaining the slow state and stretching the run out by an extra 20-30 minutes.
