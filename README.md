# Autonomous Multi-Agent Swarm for 5k-Node Kubernetes Triage

This project provides an automated, LLM-driven "CLI-Native" agent swarm designed to triage massive, 55 GB data artifacts generated by the `ci-kubernetes-e2e-gce-scale-performance-5000` test suite. 

By leveraging the `gemini` CLI, this swarm autonomously downloads ground-truth test results, streams gigabytes of logs using zero-memory Unix pipes, diagnoses deep architectural bottlenecks (like API server CPU serialization), and outputs definitive software engineering bug reports.

---

## 🚀 How to Use the Swarm

The swarm is entirely self-contained within the `gemini` CLI environment using Markdown-based system prompts. You do not need to compile any Go binaries or install custom orchestration software.

### Prerequisites
1.  **Install the Gemini CLI**: Ensure the `gemini` CLI is available in your `$PATH`.
2.  **GCP Authentication**: The swarm requires read access to the public Kubernetes CI bucket. You must authenticate and define a billing project:
    ```bash
    gcloud auth login
    gcloud config set project <your-local-devel-project>
    ```

### Execution
To launch the triage swarm, execute the CLI using the Master Orchestrator prompt:

```bash
gemini -f orchestrator-prompt.md
```

**What happens next?**
1.  **Polling**: The Orchestrator will automatically ping GCS to find the `latest-build.txt`.
2.  **Filtering**: If the build failed, it will execute a series of `gcloud storage cat | grep/tail` streams into `/tmp/k8s-triage/[BUILD_ID]/` to reduce 55 GB of raw logs into a few megabytes of dense error strings.
3.  **Delegation**: The Orchestrator will invoke the appropriate specialized sub-agents (`infra-expert`, `control-plane-expert`, `metrics-expert`).
4.  **Reporting**: The swarm will synthesize the deep-dive analysis and write a permanent bug report to `./triage-journals/[BUILD_ID]/journal.md`.

## 🏗️ Swarm Architecture (The Mapper-Reducer Pattern)

The system relies on a rigid separation of concerns to prevent LLM context-window exhaustion and host machine Out-Of-Memory (OOM) errors.

### 1. The Master Orchestrator (The "Mapper")
Located at `orchestrator-prompt.md`. The orchestrator is the only agent that knows the GCS directory structure, artifact sizes, and the shell commands required to fetch them. 
It acts as a data pipeline engineer, using native Unix pipes (`|`) to stream data without loading full files into memory. 

### 2. The Domain Experts (The "Reducers")
Located in `.gemini/agents/`. These are highly specialized sub-agents registered natively as CLI tools. They are completely ignorant of GCS paths. They simply consume the `/tmp/` artifacts passed to them by the Orchestrator, apply known error signatures, and output technical reports.

*   **`infra-expert`**: Diagnoses Google Compute Engine (GCE) provisioning locks (HTTP 400s) and teardown scripts.
*   **`control-plane-expert`**: Diagnoses API Server Priority & Fairness (APF) saturation, `http2: client connection lost` drops, and Go Panics.
*   **`metrics-expert`**: Diagnoses API Responsiveness SLO breaches, tracks "Thundering Herd" call volume variances, and correlates bottlenecks to CPU profiling (`.pprof`) data.

## 🧠 Core Triage Principles

This swarm is programmed with advanced CI/CD debugging principles to ensure it finds true root causes rather than superficial symptoms:

1.  **Ground Truth vs. Wrapper Symptoms**: The swarm will never assume a final `exit status 255` teardown crash is the root cause. It is programmed to explicitly pull `junit.xml` to determine if the core workload had already failed, proving the infrastructure timeout was merely a spurious secondary symptom.
2.  **Concurrency as a Latency Driver**: The swarm analyzes request *volume* (Count) across builds. It understands that latency spikes are rarely random; they are usually caused by a spike in concurrency saturating an underlying bottleneck.
3.  **Profiling for Proof**: The swarm does not stop at log hypotheses. It correlates anomalies to underlying Go Garbage Collection (GC) churn and serialization functions found in `.pprof` artifacts.

## 🔎 Real-World Case Study: The "Thundering Herd"

During the development of this swarm, manual engineering triage incorrectly assumed that recent 5k-node test failures were due to a GCE infrastructure deadlock (an inability to delete the `b-control-plane` load balancer during cluster teardown). 

When the autonomous swarm was executed against these runs, it exposed a radically different truth:

1.  **The Primary Failure**: The `junit.xml` proved the tests actually failed earlier due to an API Responsiveness SLO breach (99th percentile for `LIST pods` reached 59.8 seconds, far exceeding the 30s limit).
2.  **The Bottleneck**: The `metrics-expert` agent downloaded the API Server CPU `.pprof` file and proved that the server was spending 17% of its CPU cycle completely locked up trying to serialize hundreds of thousands of protobuf objects into JSON.
3.  **The "Thundering Herd"**: The swarm analyzed successful vs. failed runs and found that successful runs averaged ~380 `LIST` calls, while failed runs saw ~600. The agent deduced that memory allocation and GC churn caused internal API channels to block, forcing `client-go` connections to drop. When those connections dropped, the clients reconnected simultaneously, issuing massive unpaginated `LIST pods` calls to re-sync their caches—a classic "Thundering Herd" that completely destroyed the API Server's CPU.
4.  **The Spurious Teardown**: Because the core test aborted early due to these latency timeouts, the provisioning framework subsequently suffered a spurious GCE resource lock deadlock during cluster deletion.

The swarm automatically identified this complex, multi-layered distributed systems failure and documented it flawlessly without human intervention.
