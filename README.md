# Autonomous Multi-Agent Swarm for 5k-Node Kubernetes Triage

This project provides an automated, LLM-driven "CLI-Native" agent swarm designed to triage massive, 55 GB data artifacts generated by the `ci-kubernetes-e2e-gce-scale-performance-5000` test suite.

Instead of relying on heavy, centralized CI/CD infrastructure, this project utilizes a **"Local-First" Architecture**. It empowers performance engineers to execute highly capable triage agents directly on their local machines, using zero-memory Unix streams and dynamically loaded Agent Skills.

---

## 🎯 The Ultimate Goal (Actionable Triage)
The entire architecture exists to produce a single, actionable outcome for the performance engineer. At the end of its investigation, the swarm MUST definitively prove one of three things:
1.  **It is an Infrastructure Flake:** The failure was caused by underlying cloud or pipeline flakiness (e.g., GCE quotas, network partitions). The agent proves the core software is healthy, allowing the engineer to safely ignore the failure or click "Retest."
2.  **It is a Code Regression (And Here is the PR):** The failure is a genuine degradation in the codebase. The agent mathematically proves the regression against a baseline, diagnoses the mechanical bottleneck (e.g., lock contention), and pinpoints the exact Pull Request in the change window that introduced the flaw, allowing the engineer to instantly click "Revert."
3.  **It is an Emergent Limit or Test Change:** The failure is real, but there is no single "bad PR" in the change window. The agent proves the system hit an organic scale limit (death by a thousand cuts), a latent race condition, or that the test framework/configuration was altered, requiring a broader architectural investigation rather than a simple PR revert.

---

## 🚀 The Local-First Architecture

To prevent LLM context-window exhaustion and host machine Out-Of-Memory (OOM) errors, the swarm relies on strict procedural knowledge and data normalization.

### 1. Foundational Skills (Procedural Guardrails)
Triage logic is decoupled into atomic, portable `.skill` files that any agent can ingest:
*   **`parse-cl2-metrics`**: Provides exact JSON paths to extract P99 latencies and SLO breaches from ClusterLoader2 artifacts.
*   **`establish-perf-baseline`**: Prevents false positives by forcing the agent to mathematically prove regressions against a known-good baseline run.
*   **`classify-flakiness`**: Provides strict heuristics for differentiating between infrastructure teardown deadlocks and genuine code regressions.

### 2. Multi-Dimensional Visual Dashboards
A massive part of this project is translating complex distributed systems failures into visually undeniable proof. The agents automatically generate a "Newspaper Layout" dashboard using the [Visualization Plan](plans/visualization-plan.md).

Using the included `scripts/generate_dashboard.py`, the swarm transforms raw TSDB metrics and `.pprof` artifacts into synchronized, multi-panel Trellis graphs covering:
*   **Concurrency Surge:** Visualizing "Thundering Herd" traffic spikes.
*   **CPU Saturation:** Stacked area charts separating Garbage Collection (GC) churn from core execution locks.
*   **Storage Health:** Etcd P99 Fsync Latency line charts.

## 🛠️ How to Use the Swarm

The swarm is entirely self-contained within the `gemini` CLI environment. 

### Prerequisites
1.  **Install the Gemini CLI**: Ensure the `gemini` CLI is available in your `$PATH`.
2.  **Install Python Dependencies**: The visualization scripts require `matplotlib`, `seaborn`, and `pandas`.
3.  **GCP Authentication**: The swarm requires read access to the public Kubernetes CI bucket.
    ```bash
    gcloud auth login
    gcloud config set project <your-local-devel-project>
    ```

### Execution
To launch the triage swarm, simply invoke the CLI and point it to the failing build:

```bash
gemini "Triage build 2066566728590036992 using the local skills and generate a visual journal."
```

**What the Agent Does:**
1.  **Streams & Filters**: Uses zero-memory pipelines (`gcloud storage cat | grep`) to pull only relevant errors and JSON metrics locally.
2.  **Baselines**: Fetches a passing run to mathematically prove the anomaly.
3.  **Generates Dashboard**: Executes `generate_dashboard.py` to render the Trellis layout `.png` graphs.
4.  **Journals**: Synthesizes the findings and visual evidence into a highly readable Markdown report at `triage-journals/[BUILD_ID]/journal.md`.

## 🔎 Real-World Case Study: The "Thundering Herd"

During the development of this swarm, manual engineering triage incorrectly assumed that recent 5k-node test failures were due to a GCE infrastructure deadlock. 

When the autonomous swarm was executed against these runs, it exposed a radically different truth:

1.  **The Primary Failure**: The `junit.xml` proved the tests actually failed earlier due to an API Responsiveness SLO breach (99th percentile for `LIST pods` reached 41 seconds).
2.  **The Bottleneck**: The agent analyzed the API Server CPU `.pprof` file and proved that the server was spending 68% of its CPU cycle completely locked up in channel queues (`runtime.selectgo`) and mutexes.
3.  **The "Thundering Herd"**: The swarm analyzed successful vs. failed runs and found that successful runs averaged ~400 `LIST` calls, while the failed run saw a surge to ~3,450. The agent deduced that internal API channels blocked, forcing `client-go` connections to drop. When those connections dropped, the clients reconnected simultaneously, issuing massive unpaginated `LIST pods` calls—a classic "Thundering Herd" that completely destroyed the API Server's CPU.

The swarm automatically identified this complex failure, rendered the visual proof, and documented it flawlessly.
